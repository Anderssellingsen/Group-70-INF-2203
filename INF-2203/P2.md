# Process initiation 

![[process_initiation]]
Once the shell reads a process name, it searches for the executable process file, presumably in the memory 

*process_start* has two options to initiate a process
1) PSTART_CALL: initiate without protection?
2) PSTART_LAUNCH: initiate with protection (i.e: the separation between user space and kernel space)? 

During process_start: 
1) process's arguments (i.e argc and argv) are pushed into the stack iteratively using *pusing_args*
2) kernel's stack pointer is given to the TSS (*to restore the stack pointer after switching from user stack perhaps?*)
3) *cpu_user_start* is called. It creates an interrupt frame, set the data segment registers,  set stack pointer and returns an iret. This simulate a return from an interrupt

After iret, CRT0 will be invoked and it takes care of the rest.

# System call and exceptions

[[system call]] is one of the new concepts of this assignment. It does so by trigger an *interruption*, which is also a **relevant** concept for this assignment 

There are two system calls that are relevant for this assignments. 
1) exit (uses for exiting a process)
2) write, which is a function to write a given variable based on a given file descriptor(uses for printf)

Based on this pre-made OS, the interrupt vector for system call is 48. The process is the sender and the kernel is recipient. When the kernel receive the signal, the kernel then switch to kernel mode with the help from **task state segment**.  
 

![[Image from report.png]]

 
![[ISR for syscall]]
- The OS switches from user mode to kernel mode by getting the segment descriptors from both data segment registers, *es and ds*. Those registers are then given kernel-level
```
x86_segsel_t ds, es;

x86_get_reg("ds", ds);
x86_get_reg("es", es);

x86_set_reg("es", KSEG_KERNEL_DATA); //KSEG_KERNEL_DATA = level 0
x86_set_reg("ds", KSEG_KERNEL_DATA);
```
